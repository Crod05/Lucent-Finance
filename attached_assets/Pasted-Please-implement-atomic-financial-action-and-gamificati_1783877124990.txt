Please implement atomic financial-action and gamification writes.

This is an architecture and integrity refactor, not a feature addition.
Do not alter unrelated product behavior or add new gamification systems.

Current problem:
Some financial actions commit first and then perform XP, mission,
streak, bonus, and achievement writes in separate calls or database
transactions. A later failure can leave the financial record saved
while the related gamification state is missing or incomplete.

Core rule:
One user action must open exactly one Drizzle database transaction.
Every database read and write required to complete that action must
use the same transaction object.

Do not create nested transactions inside the action transaction.

1. Transaction creation

Within one database transaction:

- Normalize and calculate the transaction fingerprint.
- Insert the transaction or identify a duplicate safely.
- Award transaction-created XP idempotently.
- Complete the eligible daily mission, including mission XP.
- Update streak state if the mission completes.
- Evaluate and award the weekly challenge if eligible.
- Complete and award the assigned bonus mission using the created
  transaction as evidence.
- Grant First Transaction if eligible.
- Evaluate Budget Guardian through a transaction-aware evaluator.
- Commit only after every required operation succeeds.

If any required operation fails:
- Roll back the inserted transaction and all gamification writes.
- Return a clear server error.
- Do not return the created transaction to the client.

Duplicate transaction behavior must remain unchanged:
- Exactly one concurrent duplicate create may succeed.
- The other request must return 409 with duplicate=true and existingId.
- Duplicate rejection must not award XP or complete missions.

2. Bill payment

Within one database transaction:

- Lock or conditionally update the eligible unpaid bill.
- Change the bill to paid.
- Award bill-paid XP idempotently.
- Complete the eligible daily mission and related streak/weekly logic.
- Complete the assigned bonus mission using the paid bill as evidence.
- Grant Bill Slayer if eligible.
- Commit only after all required operations succeed.

Repeated or concurrent payment requests must not:
- award XP twice;
- complete a bonus twice;
- update the streak twice;
- grant an achievement twice.

If a required gamification operation fails, the bill must remain unpaid.

3. Budget review intent

Use one database transaction to:

- Verify the assigned daily mission is review_budget and is pending.
- Complete the mission.
- Award mission XP.
- Update streak and weekly challenge state.
- Evaluate Budget Guardian if this endpoint remains an intended
  evaluation trigger.

Repeated requests must remain idempotent.

4. Insights view intent

Use one database transaction to:

- Verify the assigned daily mission is check_insights and is pending.
- Complete the mission.
- Award mission XP.
- Update streak and weekly challenge state.
- Grant Insight Seeker if eligible.

Repeated requests must remain idempotent.

5. Transaction-aware helper architecture

Refactor gamification helpers so atomic routes can pass an existing
Drizzle transaction object through the complete call chain.

Create or standardize internal helpers such as:

- awardXpForEventInTx
- completeMissionIfPendingInTx
- completeBonusIfAssignedInTx
- grantAchievementIfNewInTx
- evaluateBudgetGuardianInTx
- read/get progress using the provided transaction context

Requirements:

- Atomic routes must never call a helper that opens a new transaction.
- Atomic routes must never perform related writes using the global db
  object once the route transaction has begun.
- All queries that decide eligibility must use the same transaction
  context so they see a consistent state.
- Keep standalone wrappers only where needed, and have them open a
  transaction and delegate to the InTx implementation.
- Avoid duplicating gamification logic between routes.

6. Failure-injection tests

Add tests that deliberately throw after the financial write but before
the transaction commits.

Do not add insecure production query parameters or environment switches
for failure injection. Use dependency injection, mocks, or test-only
helpers.

Required rollback tests:

Transaction creation:
- Simulate XP failure after transaction insert.
- Assert no transaction row remains.
- Assert no XP event, mission completion, bonus mission, achievement,
  streak change, or weekly challenge award remains.

Bill payment:
- Simulate failure after bill status update.
- Assert the bill remains unpaid.
- Assert no related XP or gamification writes remain.

Also test failure at a later gamification step, such as after mission
completion but before bonus or achievement processing, to prove that
earlier gamification writes also roll back.

7. Success and regression tests

Confirm:

- Normal transaction creation still produces the expected transaction,
  XP, mission, bonus, streak, weekly challenge, and achievements.
- Normal bill payment still produces the expected results.
- Duplicate transaction behavior remains 201/409 under concurrency.
- Repeated bill payment remains idempotent.
- Budget and insights intent requests cannot award twice.
- All GET and HEAD routes remain side-effect free.
- Existing evidence-reference and fingerprint integrity tests still pass.
- Existing 44 tests continue to pass in addition to the new tests.

8. Error handling

- Do not swallow a failed gamification operation and return success.
- Log the internal failure without exposing stack traces or database
  details to the client.
- Return an appropriate 500 response for unexpected atomic-action
  failures.
- Return existing 400/404/409 responses for validation, missing records,
  and duplicate conflicts.
- Send success responses only after the database transaction commits.

9. Schema and migrations

- No schema migration is expected unless the refactor genuinely requires
  a schema change.
- Do not generate an empty migration.
- If a schema change is required, explain why and add a checked-in
  incremental migration with tests.
